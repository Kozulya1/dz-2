# Домашнее задание к занятию «Резервное копирование баз данных»
## Задание 1. Резервное копирование
### 1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.
Цель: Восстановление базы данных до состояния на конец предыдущего рабочего дня. Это означает потерю данных только за текущий день, что является относительно мягким требованием к показателю RPO (Recovery Point Objective) — допустимому объёму потерянных данных.
Оптимальные варианты резервного копирования:
#### Ежедневное полное резервное копирование (Full backup).
Создание полной копии всей базы данных один раз в день.
Это самый простой и надежный способ для данной задачи. Для восстановления потребуется только одна резервная копия — самая последняя полная (созданная прошлой ночью). Восстановление будет максимально быстрым, что соответствует хорошему показателю RTO (Recovery Time Objective) — времени на восстановление.

#### Комбинация: Еженедельное полное копирование + Ежедневное дифференциальное копирование.
Полный бэкап создается раз в неделю (например, в воскресенье). В остальные дни (понедельник-суббота) создаются только дифференциальные копии, которые содержат все изменения, произошедшие с момента последней полной копии.
Для восстановления данных за предыдущий день потребуется всего два файла: полная копия за воскресенье и дифференциальная копия, созданная вчера. Это экономит место на диске по сравнению с хранением ежедневных полных копий и ускоряет процесс создания бэкапов (по сравнению с ежедневным полным).

### 1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.
Цель: Восстановление базы данных до состояния, максимально близкого к моменту сбоя (с часовой потерей данных). Это очень жесткое требование к показателю RPO и требует более сложных стратегий резервирования.
Оптимальные варианты резервного копирования:
#### Физическое резервное копирование с непрерывной архивацией WAL-журналов (Point-in-Time Recovery - PITR).
Периодически (например, раз в день) создается базовая физическая копия файлов данных. Параллельно ведется непрерывное архивирование журналов транзакций (WAL в PostgreSQL, бинарные логи в MySQL), в которые записываются все изменения базы данных.
Этот метод позволяет восстановить базу данных на любой момент времени. В случае сбоя вы восстанавливаете базовую копию, а затем "накатываете" на неё все WAL-журналы до нужного часа (за час до поломки). Это обеспечивает минимальную потерю данных.

#### Синхронная или асинхронная репликация в сочетании с резервным копированием.
Настройка реплики, которая в реальном времени или с минимальной задержкой получает все изменения с основного сервера (мастера).

Хотя репликация не является резервным копированием, она позволяет минимизировать RPO. В случае сбоя мастера, реплика может быть быстро повышена до мастера, и потеря данных будет измеряться секундами или минутами. Однако, это не отменяет необходимости делать традиционные бэкапы для защиты от логических ошибок или случайного удаления данных.

## Задание 2. PostgreSQL

Утилита pg_dump создает логический дамп базы данных. Важно помнить, что она не блокирует работу базы данных во время создания копии .

#### Базовый синтаксис:
```
pg_dump [ПАРАМЕТРЫ_ПОДКЛЮЧЕНИЯ] [ПАРАМЕТРЫ] [ИМЯ_БАЗЫ_ДАННЫХ]
```
#### Примеры команд:

Создание дампа в текстовом формате (SQL-скрипт):
Этот формат удобен для просмотра и редактирования, но медленнее для восстановления больших объемов данных .

```
pg_dump -U postgres -h localhost -p 5432 -d mydb > mydb_backup.sql
```
Создание дампа в кастомном формате:
Кастомный формат (-Fc) сжимает данные, ускоряет восстановление и позволяет выборочно восстанавливать объекты .

```
pg_dump -U postgres -h localhost -p 5432 -d mydb -Fc -f mydb_backup.dump
```

#### Способ восстановления зависит от формата созданного дампа.

Вариант 1: Восстановление из кастомного, tar или directory формата с помощью pg_restore
Этот метод рекомендуется, так как pg_restore гибок и позволяет использовать параллельное восстановление .

#### Базовый синтаксис:

````
pg_restore [ПАРАМЕТРЫ_ПОДКЛЮЧЕНИЯ] [ПАРАМЕТРЫ] [ФАЙЛ_ДАМПА]
````
Примеры команд:

Восстановление в новую базу данных:


```
pg_restore -U postgres -h localhost -p 5432 -d newdb mydb_backup.dump
```
Восстановление с очисткой существующей БД:
Параметр --clean удаляет существующие объекты перед восстановлением .
```

pg_restore -U postgres -h localhost -p 5432 -d mydb --clean mydb_backup.dump
```
Восстановление с созданием базы данных из дампа:
Параметр -C заставит pg_restore создать базу данных, имя которой указано в дампе .

```
pg_restore -U postgres -h localhost -p 5432 -d postgres -C mydb_backup.dump
```
Вариант 2: Восстановление из текстового (plain text) дампа с помощью psql
Текстовые дампы (.sql) — это обычные SQL-скрипты, поэтому их восстанавливают с помощью клиента psql .

#### Базовый синтаксис:

```
psql [ПАРАМЕТРЫ_ПОДКЛЮЧЕНИЯ] -f [ФАЙЛ_ДАМПА]
```
Пример команды:

```
psql -U postgres -h localhost -p 5432 -d mydb -f mydb_backup.sql
```

## Задание 3. MySQL
В MySQL нет одной единственной команды для инкрементного резервного копирования. Инкрементное резервное копирование в MySQL реализуется путем резервирования бинарных логов (binary logs), которые записывают все изменения данных .
Основной инструмент для работы с бинарными логами — утилита mysqlbinlog.
#### Базовый синтаксис:
```
mysqlbinlog --read-from-remote-server --host=ХОСТ --raw [ПАРАМЕТРЫ] ИМЯ_ПЕРВОГО_ЛОГ_ФАЙЛА
```
Примеры команд:

Статическое (однократное) резервирование существующих файлов логов. Можно явно перечислить все файлы или использовать опцию --to-last-log, чтобы скопировать все файлы, начиная с указанного :

```
# Копирование файлов binlog.000130, .000131, .000132
mysqlbinlog --read-from-remote-server --host=192.168.1.100 --raw binlog.000130 binlog.000131 binlog.000132
```
```
# Копирование всех файлов, начиная с binlog.000130
mysqlbinlog --read-from-remote-server --host=192.168.1.100 --raw --to-last-log binlog.000130
```
Непрерывное (live) резервирование, при котором утилита подключается к серверу и непрерывно копирует новые события по мере их появления. Это реализует практически реальное время (near-real-time) для инкрементного бэкапа :

```
mysqlbinlog --read-from-remote-server --host=192.168.1.100 --raw --stop-never binlog.000130
```
Восстановление последнего полного бэкапа (например, созданного mysqldump) :

```
mysql -u root -p < dump_file.sql
```
Применение инкрементных изменений из бинарных логов с помощью mysqlbinlog и передачи вывода в клиент mysql :

```
mysqlbinlog binlog.001002 binlog.001003 binlog.001004 | mysql -u root -p
```
